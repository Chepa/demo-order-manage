## Доменная модель заказов

- **Сущность `Order`**:
  - Идентификатор заказа.
  - Клиент (`customer_id`).
  - Статус.
  - Общая сумма заказа.
  - Коллекция позиций заказа (items).
  - Метаданные (даты создания/обновления и т.п.).

- **Сущность `OrderItem`**:
  - Идентификатор позиции.
  - Ссылка на заказ.
  - Ссылка на товар.
  - Цена на момент заказа.
  - Количество.
  - Подсчитанная сумма по позиции (price * quantity).

### Статусы заказа

Предполагаемые статусы (по текущему коду):

- `pending` — заказ создан, но ещё не подтверждён/не оплачен.
- `confirmed` — заказ подтверждён, зарезервирован товар и зафиксирована сумма.
- `shipped` — заказ отгружен.
- `completed` — заказ выполнен (доставлен и закрыт).
- `cancelled` — заказ отменён.

### Допустимые переходы статусов

Допустимые переходы (на основе текущей реализации `VALID_TRANSITIONS` модели `Order`):

- `pending` → `confirmed` или `cancelled`.
- `confirmed` → `shipped` или `cancelled`.
- `shipped` → `completed`.
- `completed` → (нет допустимых дальнейших переходов).
- `cancelled` → (нет допустимых дальнейших переходов).

### Инварианты и бизнес-правила

- Сумма заказа (`total_amount`) всегда должна быть равна сумме всех позиций:
  - \(`total_amount` = Σ (item.price * item.quantity)\).
- Для каждой позиции заказа:
  - `quantity` > 0.
  - `price` фиксируется на момент создания заказа и не зависит от последующих изменений цены товара.
- При создании заказа:
  - Для каждого товара достаточно остатка (`stock_quantity >= requested_quantity`).
  - При успешном создании заказа остатки уменьшаются на запрошенное количество.
- При отмене заказа (из допустимого состояния):
  - Остатки могут быть возвращены (в зависимости от бизнес-требований; по текущему коду можно добавить эту логику в домен).
- Переход статуса:
  - Разрешён только если новый статус входит в список допустимых переходов для текущего статуса.
  - Повторная установка того же статуса не изменяет заказ и может считаться no-op.

### Доменные операции

- Создание заказа:
  - Валидация инвариантов (наличие клиента, хотя бы одна позиция, корректные количества).
  - Проверка и резервирование остатков.
  - Подсчёт общей суммы.
  - Инициализация статуса `pending` или `confirmed` (в зависимости от бизнес-сценария).

- Добавление/изменение позиций:
  - Обновление набора позиций и перерасчёт суммы.
  - Повторная проверка остатков (если меняются количества).

- Смена статуса:
  - Проверка допустимости перехода (`canTransitionTo`).
  - Применение побочных эффектов (например, фиксация отгрузки при переходе в `shipped`).
